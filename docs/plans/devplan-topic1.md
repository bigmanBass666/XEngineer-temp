## 三、剩余风险项（需讨论）

### 3.1 🔴 火山 ASR 二进制协议实现

**具体挑战：**
- 4 字节 header：前 2 字节 payload 长度（大端序），后 2 字节预留
- 音频需要 Gzip 压缩后发送
- 需要实现完整的对话状态机（发送音频 → 接收中间结果 → 接收最终结果）
- 二遍识别模式：先快速出结果，再出精确结果

**应对策略：**
- 直接基于火山官方 Python Demo 改造（文档提供了下载链接）
- 预计工作量：6h（Python + 官方 Demo 基底 vs 从零 Node.js 实现 10-12h）
- **但如果官方 Demo 代码有问题或文档与实际 API 不一致，调试时间可能翻倍**

**讨论点：**
- 官方 Demo 是否可靠？是否有人跑通过？
- 是否需要在开发早期就做一次 ASR 连通性测试（验证 API Key + 协议可用），还是直接边做边调？

### 3.2 🔴 前后端通信架构

**问题：** Python 后端需要同时处理三种 WebSocket 连接：
1. 前端 → 后端：音频流 + 截图
2. 后端 → 火山 ASR：二进制 WebSocket
3. 后端 → 火山 TTS：HTTP 请求

**具体设计问题：**

**a) 前后端用一条 WebSocket 还是多条？**

| 方案 | 说明 | 优劣 |
|------|------|------|
| **单 WebSocket** | 前后端一条连接，用 JSON 消息类型区分（audio/image/text） | 简单，但音频流和文字消息混在一起，需要消息协议设计 |
| **双 WebSocket** | 一条传音频流（二进制），一条传控制消息（JSON） | 音频和文字分离，但前端管理两个连接 |

**我的建议：** 单 WebSocket + 消息协议。前端发送 `{type: "audio", data: base64}` 和 `{type: "image", data: base64}`，后端发送 `{type: "text", content: "..."}` 和 `{type: "audio", data: base64}`（TTS 音频）。

**b) 音频从前端传到后端的格式？**

浏览器 `getUserMedia` 采集的是原始 PCM，需要：
- 前端用 `AudioContext` 采集 16kHz 单声道 PCM
- 转换为 base64 后通过 WebSocket 发给后端
- 后端解码后 Gzip 压缩，再按火山二进制协议发给 ASR

**或者更简单的方案：** 前端直接用 `MediaRecorder` 录成 webm/opus，后端用 ffmpeg 转 PCM？但这增加了 ffmpeg 依赖。

**讨论点：**
- 音频传输格式你有什么偏好？
- 是否需要做 VAD（语音活动检测）？还是用户按住按钮说话（push-to-talk）？VAD 在前端做还是后端做？

### 3.3 🟡 火山 TTS HTTP 流式音频回传

**问题：** 火山 TTS HTTP 接口返回的是流式音频（Chunked Transfer），后端需要把音频流实时转发给前端播放。

**挑战：**
- Python 后端接收 HTTP 流式响应，同时通过 WebSocket 推送给前端
- 前端需要接收音频 chunk 并无缝播放（Web Audio API + AudioBuffer 队列）
- 如果 LLM 生成文本较慢，TTS 要等文本完成才能开始合成（延迟叠加）

**优化方案：**
- LLM 流式输出 → 积累到一句话（句号/问号）→ 立即调 TTS → 边合成边播放
- 前端音频播放队列：新音频到来时排队播放，不打断当前播放

### 3.4 🟡 Agnes API 调用方式

**问题：** Agnes Text API 是通过 HTTPS 调用的（`https://apihub.agnes-ai.com/v1/chat/completions`），Python 后端用 `requests` 或 `httpx` 调用。

**需要注意：**
- Agnes API Key 通过 `.env` 配置，和火山 API Key 一起管理
- 图片理解：截图 base64 直接放在 `image_url` 的 `url` 字段（Data URI 格式）
- 流式输出：Agnes 支持 SSE 流式，Python 用 `httpx` 的 stream 模式可以逐步获取文本
- **Agnes API 的公网可达性**：需要确认 `apihub.agnes-ai.com` 从本地网络能正常访问（不在沙箱内）

**讨论点：**
- Agnes API 是否公网可达？如果也是平台内网，那离开沙箱也用不了，这会是致命问题
- 如果 Agnes 不可用，是否有备选 LLM/VLM？（火山引擎有豆包大模型，但需要另外开通）

### 3.5 🟡 LLM 响应 + 视觉理解的协同

**工作流程：**
1. 用户说话 → ASR 转文字："这是什么？"
2. 同时截图 → base64 编码
3. 将文字 + 图片一起发给 Agnes Text → "用户问：这是什么？[图片]"
4. Agnes 返回文字回复
5. 回复发给 TTS → 音频推送给前端

**需要设计的 System Prompt：**
```
你是一个AI视觉对话助手。用户会向你展示他们摄像头中的画面，并问你问题。
请结合画面内容和用户的问题，给出准确、友好的回答。
如果用户的问题与画面无关，正常回答即可。
```

**讨论点：**
- 是否需要支持"纯语音模式"（不截图，纯文字对话）？
- 对话历史保留多少轮？每轮都带图片还是只最近一轮？

### 3.6 🟡 摄像头截图策略

| 策略 | 说明 | 复杂度 |
|------|------|--------|
| **每轮都截** | 用户说话就截图发给 VLM | 最简单 |
| **VAD 触发** | 检测到语音开始时截图 | 中等 |
| **关键词触发** | ASR 结果含视觉相关词时才截图 | 复杂 |

**我的建议：** MVP 用"每轮都截"（最简单）。成本控制靠低分辨率（640×480 JPEG quality 0.6）和上下文裁剪（只保留最近 5 轮图片）。
