## 必须修复的 5 个高风险问题

### 1. PR 拆分策略缺失（三份报告共同指出）

> 红线："仅在最后一天一次性导入所有代码 → 直接视为无效作品"。当前计划完全没有 PR 级别的规划。

**建议**：至少 25-30 个 PR，Phase 1 每1-2h一个，Phase 2 每3-4h一个。

### 2. 6月13日空白日 + PR集中在最后一天（Git审查发现）

> 所有10个PR都在6月14日合并，6月13日零commit。评委视角下这就是"尾日突击"。

**建议**：正式开发的PR必须均匀分布在72h内。

### 3. ASR 二进制协议工作量被严重低估（技术审查发现）

> 计划给4h，实际评估需要6-8h。还有 Float32→Int16 PCM格式转换、采样率对齐等隐性陷阱未提及。TTS的"流式"描述也不准确——HTTP接口返回完整base64音频，不是真正的流式chunk。

**建议**：Phase 1.4 作为最高优先级，4h不通立即启用 Web Speech API fallback。

### 4. 设计文档缺少"计划vs实际"对比结构（合规审查发现）

> 赛题要求"计划实现哪些用户故事，最终实现了哪些"+"想到了哪些成本控制技巧，实际采用了哪些"。计划只列了"计划"部分。

**建议**：在Phase 4中设计对比表格模板。

### 5. 主分支中间状态可能不可运行（Git审查发现）

> "每个PR只做一件事"与"合并后main可运行"存在冲突——Pipeline在部分Node缺失时无法端到端运行。

**建议**：引入 StubNode 模式，未实现的Node返回透传。

## 需要补充的技术细节

| 遗漏 | 影响 | 建议方案 |
|------|------|---------|
| Python WebSocket库选型 | ASR客户端需要 | 后端用FastAPI内置WS；火山ASR用 `websockets`（asyncio原生） |
| HTTP客户端库 | Agnes/TTS调用 | `httpx`（支持async+SSE） |
| VAD具体算法 | "能量阈值"过于模糊 | 推荐 `@ricky0123/vad-web`（Silero VAD的WASM封装） |
| 前端AudioWorklet→Int16转换 | PCM格式不匹配 | Float32→Int16: `int16 = float32 * 32768`，clamped |
| 采样率对齐 | 浏览器默认44100Hz, ASR要16000Hz | 前端 `new AudioContext({sampleRate: 16000})` |
| Pipeline节点通信机制 | 完全未定义 | `asyncio.Queue` |
| 错误处理策略 | 完全缺失 | 至少需要ASR重连、API失败提示、WS断连重连 |
| CORS配置 | 前后端跨域 | FastAPI加 `CORSMiddleware` |
